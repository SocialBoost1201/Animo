## 0. 業務要件（プロダクトの「あるべき」動き）

### 0.1 フロー概要

1. **管理側**がキャスト情報を事前登録する。
  **登録項目（想定）:** 本名、フリガナ、源氏名、生年月日、携帯番号、メールアドレス（任意）、LINE ID
2. 事前登録が完了すると、**キャスト**は `cast/register`（およびモバイル相当パス）から **アカウント作成** ができるようになる。
3. **キャスト側の入力項目:** 苗字、名前、源氏名、携帯番号、LINE ID（項目数は管理側より少ない）。
4. 入力後「新規登録」で登録が反映され、**ログイン画面へ誘導**される想定。
5. 登録した携帯番号で SMS 認証コードを受け取り、入力後 **キャストダッシュボード** に入る。

### 0.2 項目設計の方針（要件として確定したいこと）


| 方針            | 内容                                                                                                             |
| ------------- | -------------------------------------------------------------------------------------------------------------- |
| **マスタの「正」**   | **管理側の登録データ**をソースオブトゥルースとする。                                                                                   |
| **キャスト画面の項目** | **少ない方（上記5項目）に寄せる**方針でよい。管理側と画面項目を機械的に同一にする必要はない。                                                              |
| **照合のためのルール** | 管理側のどの列と、キャストのどの入力を突き合わせるかを **一文で固定**する（D の成果物）。例: 本名 ↔ 苗字+名前の結合、`casts.name_kana` ↔ キャストの源氏名入力（サーバー実装と一致させる）。 |


### 0.3 本計画書のスコープに含めないもの（別タスク）


| 内容                     | 備考                                                                                                  |
| ---------------------- | --------------------------------------------------------------------------------------------------- |
| **14日間・SMS なしでの再ログイン** | 仕様として望ましいが **未実装／不調の報告あり**。Cookie・middleware・OTP まわりの **別計画・別 PR** とする。本書は **新規登録とエラー契約** にフォーカスする。 |

## 1. 「契約ファイル」とは（用語）

**契約ファイル**とは、法律文書ではなく **開発・運用で共有する「エラーと対応の一覧表」** を指す。

- **何を載せるか:** サーバーが返す `code`、ユーザー向けメッセージ、画面タイトル（`errorTitleMap`）、意味、一次対応の役割（A〜F）。
- **なぜ必要か:** UI・サーバー・運用で「同じエラー」を別の言い方で扱わないための **単一の参照元**。
- **置き場所（案）:** `docs/agent-system/cast-register-errors.md`（新規作成予定）

コード上の「契約」は `**castRegister` の戻り値の `code` とメッセージ**。契約ファイルはその **人間可読の写し + 運用メモ**。

## 2. 目的とスコープ

### 2.1 目的

- `castRegister` が返しうる **すべての結果コード（`code`）** を、**単一の契約**として UI・運用・監視で共有する。
- **データ照合の仕様（D）を先に固定**し、フロント（B）とサーバー（C）の認識ズレによる `NO_MATCH` を構造的に減らす。
- **運用（E）・監視（F）** で設定不備・Auth 不整合・障害を切り分け可能にする。

### 2.2 スコープ内

- エラーコード一覧と契約ドキュメント（上記パス想定）
- サーバー：`code` の網羅・細分化、ログ・Auth 分岐、LINE 更新の成否
- フロント：クライアント検証、`errorTitleMap` 完全対応、文言・ヘルプ（照合ルールに沿った説明）（A+B）
- **データ仕様書（D）:** 管理マスタ列 ↔ キャスト入力 ↔ DB 照合キーの対応表
- 運用手順・監視方針のドキュメント（E+F）※機密を書かない範囲

### 2.3 スコープ外（別計画・要承認）

- RLS や認可の緩和
- 本番シークレットの値の記載
- アプリ本体の無関係な画面の UI 刷新
- **14日再ログイン（SMS 省略）の実装・修正**（§0.3）

## 3. 役割分担（サブエージェント／担当領域）


| 役割                             | 主な責務                                   | 今回のエラー対策との対応                                                 |
| ------------------------------ | -------------------------------------- | ------------------------------------------------------------ |
| **A. プロダクト / UX**              | フォーム文言・ラベル・ヘルプ・エラー表示の一貫性               | `VALIDATION_INCOMPLETE` / `NO_MATCH` の減少、**事前登録との表記合わせ**の明文化 |
| **B. フロント（登録画面）**              | クライアント検証、二重送信、`errorTitleMap`、フィールド別表示 | `INVALID_PHONE`、全 `code` の表示網羅、体験の揃い                         |
| **C. サーバー（`castRegister` 周辺）** | バリデーション、`code` 細分化、Auth/DB エラー解釈       | `AUTH_`*、`MATCH_*`、`ROLE_*`、`CAST_*`、設定系の分離                  |
| **D. データ / ドメイン**              | **管理マスタとキャスト入力の照合ルール**の単一の「正」          | `NO_MATCH` / `MULTIPLE_MATCHES` の根本対策                        |
| **E. 運用 / SRE**                | env チェックリスト、Auth の孤児ユーザー整理、重複マスタ修正手順   | `UNEXPECTED`（設定）、`AUTH_RECOVERY_`*、インフラ障害時                   |
| **F. 品質 / 監視**                 | ログ方針、アラート、PII マスク、再発時の調査手順             | `MATCH_LOOKUP_FAILED`、catch `UNEXPECTED`、レート制限の検知            |

## 4. 依存順序（「全部が動き出す」順）

以下の順でタスクを切る。**仕様（D）が固まる前に B/C の実装を広げると手戻りが増える。**

```text
D（データ仕様・照合ルールの確定：管理列 ↔ キャスト入力 ↔ DB）
  → C（返却 code・メッセージ・サーバー挙動）
    → A + B（ラベル・ヘルプ・クライアント検証・errorTitleMap）
      → E + F（デプロイ検証手順・ログ・アラート・運用 Runbook）
```

### 要点

1. **先に D（データ仕様）を固定する**
  管理マスタとキャスト5項目の対応が決まらないと、`NO_MATCH` が続く。
2. **C で返す `code` を揃えてから B で拾う**
  タイトル・フィールド別エラー・リトライ文言はすべて `**code` を契約**として繋ぐ。

## 6. 契約ファイル（必須アウトプット）

**パス案:** `docs/agent-system/cast-register-errors.md`（新規作成）

**最低限の表項目:**


| 列           | 説明                       |
| ----------- | ------------------------ |
| `code`      | サーバーが返す識別子（省略時の扱いも記載）    |
| 意味          | 開発者向け説明                  |
| ユーザー向けメッセージ | そのまま表示するか、マップするか         |
| UI タイトル     | `errorTitleMap` の値       |
| 主原因         | ユーザー起因 / データ不整合 / インフラ 等 |
| 対応責任        | A〜F のどれが一次対応か            |


**M2 で追加推奨:** 「このエラー時にユーザーが確認すべきマスタ項目」（D と連動）

M1 では **この表と `castRegister` の分岐が 1:1** になるまでレビューする。

## 7. サブエージェント運用のコツ

- **単一の契約ファイル**を正とし、**コード変更は表の更新と同一 PR／同一リリース単位**を推奨。
- **依存は「仕様 → サーバー → UI → 運用」**。並列化できるのは、D が確定した後の **B の表示調整** と **F の監視設計** など。
- **PII**（電話・本名）はログに出さない方針を F と C で共有。
